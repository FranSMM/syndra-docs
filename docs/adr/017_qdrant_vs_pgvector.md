# ADR 017: Selection of Qdrant over pgvector for Vector Search

**Status:** Accepted

## Context
As the Syndra Data Engine expands its NLP capabilities, we require a robust system for storing and querying high-dimensional vector embeddings generated by our SentenceTransformer models for semantic search functionalities. The two primary candidates under consideration were `pgvector` (a Postgres extension) and Qdrant (a dedicated vector database).

## Decision
**Qdrant** was selected as the primary vector store for the Gold layer, rather than utilizing `pgvector` within our existing PostgreSQL instance.

## Reasoning
1. **Separation of Concerns:** PostgreSQL excels at managing our relational, structured data within the Bronze and Silver layers. Offloading the vector workloads to Qdrant allows a clean boundary where Qdrant serves exclusively as the semantic search engine (Gold layer).
2. **Scalability:** By decoupling the vector store from the relational database, we can scale our vector search infrastructure independently of our transactional Postgres database if our embedding load increases significantly.
3. **Native Vector Features:** Qdrant is built from the ground up as a vector database. It includes out-of-the-box Hierarchical Navigable Small World (HNSW) indexes, highly optimized payload filtering, and an API specifically designed for semantic search, which outperforms the more limited capabilities of `pgvector`.

## Consequences
- Adoption of polyglot persistence (SQL + Vector NoSQL).
- Increased infrastructure footprint requiring a dedicated Qdrant container.
- High-performance, scalable semantic search capabilities.
